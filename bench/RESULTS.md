# JS vs Wasm trace比較 — 結果と発見

Environment: macOS (Apple Silicon) / Node v20.19.0 / 20 images × 1280×720
Fixture: `bench/fixtures/` generated by `./generate.sh`
Branches / patches:
- `otel` branch of reg-cli
- `image-diff-rs` patched to add `decode_*` / `compare_pixels` / `encode_webp_ffi` spans
- `js/worker.ts` / `js/entry.ts` / `js/index.ts` patched to record JS-bridge spans

## 1st run — tracing shutdown includes 2s sleep (original code)

| version | wall-clock |
|---------|------------|
| JS      | 1.58 s |
| Wasm    | **3.30 s** (2.1× slower) |

Rust-reported core: 866ms. "missing" ≈ 2430ms — assumed to be wasm instantiate /
Worker spawn / boundary copy.

## 2nd run — detailed spans added across JS bridge and image-diff-rs

With the full instrumentation in place, we could see where the 2.4s was going.
Answer: **it was a `setTimeout(2000ms)` inside `shutdownTracing()`** added as a
workaround to give OTLP exports time to flush. See
[`js/tracing.ts`](../js/tracing.ts) — the sleep is gone now.

## 3rd run — after removing the 2s sleep

| version | wall-clock | ratio |
|---------|------------|-------|
| JS      | 0.81 s | baseline |
| Wasm    | **0.49 s** | **0.60× (1.65× faster)** |

Rust-reported core: 311ms (of ~490ms wall-clock).

Summed CPU across 4 parallel workers:
- decode (PNG): **77 ms** (cheap)
- compare_pixels (pixelmatch): **497 ms**
- encode (WebP FFI): **342 ms**
- Rust overhead (span bookkeeping, io): ~80 ms

Wall-clock 311 ms ≈ main wasm work (282 ms) + overlapping worker time.

## 重要な示唆

1. **最大の遅延要因は OTel の 2s sleep だった**
   - "wasm is slow" ではなく "wasm + trace workaround is slow"
   - 発表でも素直にここを言うべき ("span を貼って初めて分かった")

2. **sleep を抜くと Wasm は JS より速い**
   - 0.49 s vs 0.81 s
   - 想定していた thread 起動 / PNG streaming は支配的ではなかった

3. **PNG decode は 77 ms しかかかっていない**
   - 仮説2（streaming decode しないと遅い）は、この規模では成り立たない
   - 旧 pngjs との比較で streaming の優位がどこまで効くかは、**巨大 PNG 単体** で再測すべき

4. **pixelmatch が支配項 (497ms)**
   - SIMD / 改良アルゴリズムで伸ばすならここ
   - 今後の最適化候補

5. **encode_webp_ffi (342ms)**
   - libwebp の C コード。SIMD 有効化 / emscripten ネイティブ実装との差分を測る余地

6. **wasm compile / instantiate は合計 ~22ms × 4**
   - 無視できない (~90ms 相当のシリアル時間) が、支配項ではない
   - Worker pool で使い回せば更に削れる

## Reproduce

```sh
# 1. Jaeger
docker run --rm -d --name reg-cli-jaeger \
  -p 4318:4318 -p 16686:16686 jaegertracing/jaeger:latest

# 2. fixtures
cd bench && ./generate.sh

# 3. build
cd .. && npm i && npm run build                    # JS
./scripts/build-wasm.sh                            # wasm (uses /tmp/image-diff-rs patch)
cd js && pnpm i && pnpm build && cd ..             # wasm bridge

# 4. benchmark
cd bench && OTEL_ENABLED=true ./run.sh

# 5. export & visualize
./export-traces.sh            # writes out/traces.json
cd .. && python3 -m http.server 8766 --directory bench
open http://localhost:8766/viz/
```
